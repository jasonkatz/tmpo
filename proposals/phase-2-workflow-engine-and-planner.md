# Proposal: Phase 2 — Workflow Engine & Planner Agent

## Summary

Build the workflow engine that picks up pending workflows and drives them through the pipeline, starting with the plan step. After this phase, running `cadence run` creates a workflow, the engine transitions it to `running`, the planner agent reads the repo and generates a structured proposal, and the user can watch progress in real-time via SSE. This is the first phase where an agent actually executes, proving the core orchestration loop works end-to-end.

## Acceptance Criteria

### Workflow Engine

1. A workflow created via `POST /v1/workflows` (status: `pending`) is automatically picked up by the engine. `GET /v1/workflows/:id` shows the status transition to `running` and a `steps` array for iteration 0 containing seven steps (plan, dev, ci, review, e2e, e2e_verify, signoff) all with status `pending`.

2. While the plan step executes, `GET /v1/workflows/:id/steps` shows the plan step with status `running`, a non-null `started_at`, and all other steps still `pending`. After the plan step completes, it shows status `passed` with both `started_at` and `finished_at` set.

3. After the plan step passes, `GET /v1/workflows/:id` returns a non-null `proposal` field containing the structured proposal generated by the planner agent. The proposal contains a Summary, Acceptance Criteria, and Technical Considerations section.

4. If the plan step fails (agent error or timeout), `GET /v1/workflows/:id` shows status `failed` with a non-null `error` field describing the failure.

5. After the plan step passes, the engine stops — subsequent steps remain `pending`. `GET /v1/workflows/:id/steps` confirms only the plan step has `passed` status.

6. `POST /v1/workflows/:id/cancel` during plan step execution causes the workflow to reach `cancelled` status after the current step finishes. No further steps transition to `running`.

7. A workflow whose `iteration` exceeds `max_iters` transitions to `failed`. `GET /v1/workflows/:id` shows an error message indicating the iteration limit was reached.

### Agent Observability

8. After the plan step runs, `GET /v1/workflows/:id/runs` returns at least one run with `agent_role: "planner"`, a non-null `prompt`, a non-null `response`, an `exit_code`, and a `duration_secs` value.

9. If the plan agent times out (exceeds the configured timeout, default 300 seconds), the step fails and `GET /v1/workflows/:id/runs` shows the run with a non-zero `exit_code`.

### Real-Time Progress

10. `GET /v1/workflows/:id/events` returns a `text/event-stream` response. When the plan step transitions from `pending` → `running` → `passed`, each transition is emitted as an SSE event containing the updated step as JSON.

11. When the workflow reaches a terminal state (`complete`, `failed`, `cancelled`), the SSE stream emits a final event and the connection closes.

### CLI

12. `cadence run --task "..." --repo owner/repo` streams step transitions to stdout as they happen (e.g., "plan: running...", "plan: passed"). The command blocks until the workflow reaches a terminal state or the user interrupts with Ctrl+C.

13. `cadence proposal <workflow-id>` prints the proposal text to stdout. Returns an error if the workflow has no proposal yet.

### Web Client

14. The workflow detail page (`/workflows/:id`) shows: the full task description, the proposal (rendered as markdown), the current status, and a step timeline showing each step's status with timing. Step transitions appear in real-time without manual page refresh.

## Technical Considerations

- **Agent runtime**: The product brief specifies pi agent core as the runtime for spawning and managing agents. The server should integrate this as a dependency for agent lifecycle management. The planner agent uses read-only tools (Bash, Read, Glob, Grep).
- **Process isolation**: The planner agent needs access to the target repository. The server should clone or create a worktree of the target repo in a temporary directory for the agent to read. The user's stored GitHub token (from Phase 1 settings) is used for cloning.
- **SSE implementation**: Use Express response streaming for SSE. The server should maintain a registry of active SSE connections per workflow and fan-out events to all connected clients.
- **Engine loop**: The engine can be a simple polling loop or event-driven. It should run in-process with the server for now (no separate worker). It processes one workflow at a time.
- **Graceful shutdown**: The server should handle SIGTERM by finishing the current step of any running workflow before shutting down, to avoid orphaned agent processes.

## Out of Scope

- **Dev agent execution** — The engine stops after the plan step. Dev, CI, review, E2E, and signoff are subsequent phases.
- **Regression loop** — No iteration cycling yet since only the plan step runs.
- **PR creation** — No GitHub PR integration.
- **Multi-user concurrency** — The engine handles one workflow at a time.
- **Run logs CLI command** — `cadence logs` is a later phase.
- **New workflow form in web client** — Creating workflows via the web is a later phase.
